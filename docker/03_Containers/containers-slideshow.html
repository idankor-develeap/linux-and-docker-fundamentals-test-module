Docker Containers

Lifecycle, ports, volumes, and environment variables

Containers are ephemeral by default — everything written inside is lost when the container is removed. This lesson covers how to run, manage, and persist data in containers.

Concept

Containers Are Ephemeral

  • A container's filesystem is isolated from the host
  • Any files written inside are lost when you run docker rm
  • The image itself is never modified — containers are disposable instances
  • This is a feature, not a bug — makes containers stateless and reproducible

To persist data, use volumes or bind mounts (covered in slide 5).

docker run -d --name web nginx   # create + start
docker stop web                  # Running → Stopped
docker start web                 # Stopped → Running
docker rm web                    # Stopped → Removed
docker rm -f web                 # force-remove while running
Networking

Port Mapping

Containers are isolated — no port is reachable from the host unless explicitly mapped.

docker run -d -p 8080:80 --name web nginx
#               ↑    ↑
#           host  container
Host port 8080

The port you open in your browser: http://localhost:8080

Container port 80

The port nginx listens on inside its isolated network

Storage

Persisting Data with Volumes

Bind Mount

A host directory is mounted into the container. Changes are visible on both sides instantly.

-v /host/path:/container/path

Best for: development (live reload)

Named Volume

Docker manages the storage path. Survives docker rm and is portable.

-v mydata:/data

Best for: production data

Configuration

Environment Variables

Pass configuration into containers at runtime — no need to bake secrets into the image.

docker run \
  -e DB_HOST=db.internal \
  -e DB_PORT=5432 \
  -e DB_PASS=secret \
  myapp:1.0

# Or load from a file:
docker run --env-file .env myapp:1.0
  • Never hard-code credentials in a Dockerfile
  • Use --env-file .env for multiple variables
  • Secrets at scale: use Docker Secrets or a vault (e.g. AWS Secrets Manager)
Summary

Key Takeaways

  • Containers are ephemeral — data is lost on docker rm
  • Use named volumes for production data, bind mounts for dev
  • Ports must be explicitly mapped: -p host:container
  • Use environment variables for runtime configuration
  • docker exec -it <name> bash — your best debugging tool
